# Agent 决策助手

## 技能描述
分析用户需求，推荐最合适且最经济的 Agent 使用方案。

## 使用场景
- 用户提出开发需求，不确定应该用哪些 Agent
- 需要评估开发成本（Agent 调用次数）
- 想要省钱，寻找最优方案

## 工作流程

### 第一步：分析需求
仔细阅读用户需求，识别：
1. **需求复杂度**：简单/中等/复杂
2. **涉及范围**：仅后端/仅前端/全栈
3. **是否新增功能**：新增/修改现有
4. **是否涉及数据库**：新建表/修改表/不涉及

### 第二步：推荐方案

根据分析结果，推荐以下方案之一：

#### 🟢 极简方案（成本：0 个 Agent）
**适用条件：**
- ✅ 简单的代码修改
- ✅ 照搬已有功能
- ✅ 需求非常明确

**推荐操作：**
直接用 Claude 处理，给出明确指令，参考已有代码

**示例：**
```
需求：新增商品的 description 字段
方案：直接对话
"在 Product 实体类添加 description 字段（String），
同步修改 ProductDTO 和 CreateProductRequest，
前端 types/api.d.ts 也要加这个字段"
```

---

#### 🟡 轻量级方案（成本：1-3 个 Agent）
**适用条件：**
- ✅ 简单的 CRUD 功能
- ✅ 表结构简单（参考已有表）
- ✅ 需求明确，不需要梳理

**推荐流程：**
```
选项 A：【架构师】→ 直接用 Claude → 【全栈联调专家】
（适合新建表 + CRUD）

选项 B：【全栈联调专家】
（适合修改接口定义）

选项 C：【后端专家】或【前端专家】
（只做一端）
```

**示例：**
```
需求：新增订单管理（CRUD）
方案：
1. 启动【架构师】设计 order 表
2. 直接对话 Claude 写代码（参考 UserController）
3. 启动【全栈联调专家】检查前后端类型同步
成本：2 个 Agent
```

---

#### 🟠 标准方案（成本：3-4 个 Agent）
**适用条件：**
- ✅ 中等复杂度功能
- ✅ 涉及多表关联
- ✅ 需要缓存或消息队列
- ⚠️ 需求基本明确（可跳过产品经理）

**推荐流程：**
```
【架构师】→ 【后端专家】或【前端专家】→ 【全栈联调专家】
```

**示例：**
```
需求：商品分类管理（树形结构，需要缓存）
方案：
1. 启动【架构师】设计表结构 + 缓存策略
2. 启动【后端专家】实现接口（含 Redis 缓存）
3. 启动【前端专家】实现树形选择器
4. 启动【全栈联调专家】检查对接
成本：4 个 Agent
```

---

#### 🔴 完整方案（成本：5 个 Agent）
**适用条件：**
- ✅ 复杂业务功能
- ✅ 需求需要梳理拆解
- ✅ 涉及多服务协作
- ✅ 核心功能，质量要求高

**推荐流程：**
```
【产品经理】→ 【架构师】→ 【后端专家】+【前端专家】→ 【全栈联调专家】
```

**示例：**
```
需求：用户积分系统（签到、消费、兑换、记录）
方案：完整流程
1. 启动【产品经理】梳理需求和 API
2. 启动【架构师】设计多表 + 缓存 + 消息队列
3. 启动【后端专家】实现积分服务
4. 启动【前端专家】实现积分管理页面
5. 启动【全栈联调专家】检查对接
成本：5 个 Agent（必要成本，不要省）
```

---

### 第三步：批量优化建议

如果用户有多个相似需求，提示：

```
检测到多个相似任务，建议批量处理以降低成本：

示例：
❌ 不好：分别为 User、Product、Order 各启动一轮 Agent（成本：15 次）
✅ 好：一次性处理 3 个模块（成本：5 次）

具体做法：
1. 【架构师】一次性设计 3 张表
2. 【后端专家】一次性写 3 个 Controller
3. 【前端专家】一次性写 3 个页面
4. 【全栈联调专家】统一检查
```

## 示例对话

### 用户输入：
```
"我想新增一个商品管理功能，包括商品的增删改查"
```

### Agent 决策助手输出：
```markdown
## 需求分析

**需求描述：** 新增商品管理 CRUD 功能

**复杂度评估：** 简单

**涉及范围：** 全栈（前后端）

**数据库变更：** 新建表（商品表）

---

## 推荐方案

### 🟡 轻量级方案

**推荐流程：**
【架构师】→ 直接用 Claude 写代码 → 【全栈联调专家】

**预估成本：** 2 个 Agent

**具体步骤：**
1. 启动【架构师】Agent，设计 product 表
   - 字段：id, name, price, stock, description, created_at, updated_at, is_deleted
2. 直接对话 Claude（不启动 Agent）：
   "参考 UserController，写一个 ProductController，CRUD 接口"
   "参考 UserList.vue，写一个 ProductList.vue"
3. 启动【全栈联调专家】检查前后端类型是否同步

**节省成本技巧：**
- 表结构简单，参考 user 表，可以自己写 DDL（省掉【架构师】）
- 功能照搬 User 模块，直接用 Claude 即可（省掉【后端专家】和【前端专家】）

---

## 替代方案

### 方案 B：极简方案（成本：0 个 Agent）

**适用条件：** 你对数据库设计有把握

**操作：**
1. 自己设计表结构（参考 user 表）
2. 直接对话 Claude：
   "参考 UserController 和 UserList.vue，帮我实现 ProductController 和 ProductList.vue，
    表结构：id, name, price, stock, description, created_at, updated_at, is_deleted"
3. 手动检查前后端类型是否一致

**成本：** 0 个 Agent

---

## 注意事项

- 商品表可能需要 category_id（分类），确认是否需要
- 如果涉及图片上传，需要集成 file-service
- price 字段建议使用 DECIMAL(10,2)
```

## 判断标准速查表

| 特征 | 极简方案 | 轻量级方案 | 标准方案 | 完整方案 |
|------|---------|-----------|---------|---------|
| 需求复杂度 | 极简单 | 简单 | 中等 | 复杂 |
| 是否新建表 | 否 | 是（简单） | 是（多表） | 是（复杂关联） |
| 是否需要缓存/MQ | 否 | 否 | 是 | 是 |
| 需求是否明确 | 是 | 是 | 是 | 否（需要梳理） |
| 成本（Agent） | 0 | 1-3 | 3-4 | 5 |
